# 07. Context Compact — 잊는 게 아니다, 치워두는 것이다

> **이전 이야기에서**: 지민은 MCPs로 외부 서비스를 연결하고, Security로 보안 경계를 설정했다.
> 도구, 지식, 컨텍스트, 권한 — 네 레이어가 모두 작동하기 시작했다.
> Stripe 결제 연동에 2시간을 쏟았다. 그리고 Claude의 응답이 느려지기 시작했다...

> *"Context will fill up; you need a way to make room"*
> — Harness Engineering

---

## "Claude가 느려졌다"

Stripe 결제 연동 작업 2시간째. 지민은 이상함을 느꼈다.

Claude의 응답이 느려졌다. 3초 걸리던 대답이 8초가 됐다.
더 이상한 건 **품질**이었다. 방금 읽은 파일 내용을 잊어버린 것 같았다.

```
지민: 현재 컨텍스트 상태는?
Claude: "현재 사용량: 87%"
```

87%. 30개 파일을 읽고, MCP 툴을 20번 호출하고, 대화 30턴을 나누는 동안 — messages 배열이 한계에 다가가고 있었다.

그런데 근본적인 물리적 제약이 있었다.

Claude의 **context window는 유한하다.**

대화 내용, 파일 읽기 결과, 커맨드 출력... 이 모든 것이 messages 배열에 쌓인다.

```
messages = [
    { role: "user", content: "인증 시스템 구현해줘" },
    { role: "assistant", content: "읽어볼게요..." },
    { role: "user", content: "[tool_result] src/auth.ts 내용 (2000줄)..." },  ← 8000 토큰
    { role: "assistant", content: "..." },
    { role: "user", content: "[tool_result] tests/auth.test.ts (500줄)..." },  ← 2000 토큰
    { role: "assistant", content: "..." },
    { role: "user", content: "[tool_result] npm test 출력..." },               ← 1000 토큰
    ...
    [30개 파일 * 평균 3000토큰 = 90,000 토큰]
    [+ 대화 내용 10,000 토큰]
    [= 100,000 토큰... 한계 접근]
]
```

대형 코드베이스를 다루거나, 오래 걸리는 작업을 하면 context가 꽉 찼다.
**Claude가 더 이상 생각할 공간이 없었다.**

## 토큰 비용 실측표

컨텍스트가 왜 빨리 차는지 이해하려면 실제 토큰 비용을 알아야 한다.

| 항목 | 대략적 토큰 수 |
|------|--------------|
| CLAUDE.md (~50줄) | ~500 토큰 |
| Rules 파일 1개 (~50줄) | ~400–600 토큰 |
| Skill 파일 1개 (~30줄) | ~300–500 토큰 |
| 파일 읽기 (100줄 소스코드) | ~400–800 토큰 |
| 파일 읽기 (500줄 소스코드) | ~2,000–4,000 토큰 |
| Bash 출력 (짧은 명령) | ~100–300 토큰 |
| `npm test` 전체 출력 | ~500–2,000 토큰 |
| `git diff` (중간 크기 변경) | ~1,000–3,000 토큰 |
| 에이전트 요약 반환 | ~200–500 토큰 |
| 사용자 메시지 1개 | ~50–200 토큰 |
| Claude 응답 1개 | ~200–1,000 토큰 |

실제 시나리오:
```
파일 탐색 20개 * 평균 2,000 토큰   = 40,000 토큰
대화 20턴 * 평균 500 토큰           = 10,000 토큰
Rules 7개 * 평균 500 토큰           =  3,500 토큰
─────────────────────────────────────────────────
총계                                = 53,500 토큰
                                      ↑ 이미 위험 구간
```

Claude Code의 컨텍스트 한계: 약 200,000 토큰.
하지만 80% (160,000토큰) 넘으면 **집중력이 떨어지기 시작한다.**
50% 이하에서 유지하는 것이 최선이다.

## 해결책: 3층 압축 전략

Anthropic은 context 관리를 **3단계 압축 레이어**로 해결했다.

```
모든 턴:
┌─────────────────────────────┐
│  새로운 tool_result 추가     │
└──────────────┬──────────────┘
               ↓
[ Layer 1: Micro Compact ] — 조용히, 매 턴마다
  오래된 tool_result를 플레이스홀더로 교체
  "[이전에 src/auth.ts를 읽었음]"
               ↓
[ 토큰 > 임계값? ]
  아니오 → 계속
  예 ↓
[ Layer 2: Auto Compact ] — 자동으로
  전체 대화를 .transcripts/에 저장 (영구 보존)
  LLM이 핵심 내용을 요약
  모든 메시지를 [요약]으로 교체
               ↓
[ Layer 3: Manual Compact ] — /compact 명령으로
  사용자가 수동으로 같은 과정을 실행
```

핵심 원칙: **기록은 디스크에 영구 보존, 활성 컨텍스트만 압축.**
"잊는" 게 아니다. "잠시 치워두는" 것이다.

## Layer 1: Micro Compact — 조용한 청소부

오래된 tool_result는 대부분 이미 "소비됐다".
파일을 읽었고, Claude는 그 내용을 바탕으로 이미 코드를 작성했다.
그 원본 파일 내용 4000토큰을 계속 들고 다닐 필요가 없다.

**매 턴마다 조용히 실행된다.** 사용자에게 보이지 않지만, 시스템이 오래된 tool_result를 플레이스홀더로 교체한다:

```
교체 전 (4000 토큰):
  messages[3] = {
    role: "user",
    content: [{ type: "tool_result", tool_use_id: "read_1",
                content: "import jwt from 'jsonwebtoken';\n\nexport function verifyToken(token) {\n  ...(500줄)..." }]
  }

교체 후 (6 토큰):
  messages[3] = {
    role: "user",
    content: [{ type: "tool_result", tool_use_id: "read_1",
                content: "[Previous: read src/auth.ts — 500 lines]" }]
  }
```

이 교체는 **매 턴 시작 시** 자동으로 일어난다:

```
Micro Compact 알고리즘:

매 턴 시작:
  for msg in messages:
    if msg.age > 3턴 AND msg.type == "tool_result":
      msg.content = "[Previous: {tool_name} {파일명}]"
      freed_tokens += original_size - placeholder_size
```

Claude는 "내가 이 파일을 읽었었다"는 사실만 기억하면 된다.
내용이 다시 필요하면 다시 읽으면 된다.

> 이것이 Layer 1이 "조용한"인 이유다. 사용자가 의식하지 못하는 사이에, 시스템이 컨텍스트를 깨끗하게 유지한다.

## Layer 2: Auto Compact — 자동 요약

토큰이 50,000을 넘으면 자동으로 실행된다.

**핵심: 트랜스크립트 아카이빙이 먼저다.** 압축하기 전에 전체 대화를 디스크에 영구 보존한다.

```
Auto Compact 절차:

Step 1: 전체 대화를 디스크에 저장 (손실 없음)
  → .transcripts/transcript_2026-03-22T10-30.jsonl
  → 모든 메시지, tool_result, 에러 포함
  → 이 파일은 삭제하지 않는다 — 영구 아카이브

Step 2: LLM에게 요약 요청
  → "이 대화를 요약해줘. 완료된 것, 진행 중인 것, 다음 할 것 포함."
  → LLM이 100,000 토큰의 대화를 ~2,000 토큰 요약으로 압축

Step 3: 요약으로 대화 교체
  messages = [
    { role: "user",      content: "[압축됨]\n\n완료: 인증 API 구현...\n진행 중: 테스트 작성..." },
    { role: "assistant", content: "이해했습니다. 계속하겠습니다." }
  ]
```

100,000 토큰이 2,000 토큰으로 줄었다.
Claude는 전체 컨텍스트를 잃지 않고 계속 일할 수 있다.

`.transcripts/` 디렉토리는 시간여행의 문이다. 언제든 과거 세션의 전체 대화를 복원할 수 있다. "잊는 게 아니라 치워두는 것"이라는 말의 기술적 구현이 바로 이것이다.

## Layer 3: /compact — 전략적 타이밍

자동 압축은 토큰이 임계값을 넘었을 때만 발생한다.
하지만 **전략적으로 압축하는 것**이 더 좋을 때가 있다.

새로운 기능을 시작하기 전, 큰 리팩토링 전...
"이전 작업의 노이즈 없이 깨끗한 컨텍스트로 시작하고 싶다."

`/compact` 명령이 이걸 가능하게 했다.

Claude Code의 추천 패턴:
```
~ 50번 툴 콜 후 → /compact 권장
새로운 기능 단계 시작 전 → /compact
컨텍스트 마지막 20%에서 대규모 리팩토링 → 하지 말 것
```

### /compact 타이밍 결정 흐름도

```
지금 /compact 해야 할까?
        │
        ▼
[컨텍스트 사용률 확인]
        │
  80% 이상? ──────────────────────────────────→ 즉시 /compact
        │
  50~80%?
        │
        ▼
[작업 전환 여부 확인]
        │
  새 기능 시작하려고? ─────────────────────────→ /compact 권장
        │
  같은 작업 계속? → 아직 괜찮음
        │
        ▼
[툴 콜 횟수 확인]
        │
  50회 이상? ─────────────────────────────────→ /compact 권장
        │
  50회 미만? → 계속 진행
        │
        ▼
[현재 컨텍스트 가치 평가]
        │
  방금 읽은 파일이 앞으로도 필요한가?
        │
  예 → 지금은 /compact 하지 말 것 (다시 읽어야 하므로)
  아니오 → /compact 고려
```

실전 팁: `/compact` 스킬을 쓰면 현재 상태를 분석해서 권장 여부를 알려준다.

```
/compact          → 상태 분석 후 권장 여부 알려줌
/compact force    → 즉시 실행
/compact check    → 현재 상태만 보고
```

## 세션을 넘어서: HANDOFF.md와 세션 관리

/compact는 **세션 내** 압축이다.
그런데 세션 자체가 끝나면? 컨텍스트가 완전히 소진됐거나, 내일 이어서 해야 할 때?

### HANDOFF.md 패턴

세션 간 인계 문서를 만든다:

```markdown
# HANDOFF.md

## 목표
OAuth 2.0 + PKCE 인증 시스템 구현

## 완료된 것
- ✅ Supabase Auth 설정
- ✅ 로그인/회원가입 API 라우트
- ✅ JWT 토큰 갱신 미들웨어

## 진행 중
- 🔄 소셜 로그인 (Google, GitHub)

## 시도했지만 실패한 것
- ❌ next-auth → Supabase와 충돌, 제거함
- ❌ 쿠키 기반 세션 → CSR에서 문제, JWT로 전환

## 다음 단계
1. Google OAuth provider 연동
2. 콜백 라우트 구현
3. 테스트 작성
```

**HANDOFF.md는 자동 생성이 아니다.** 메커니즘을 이해하자:

```
작성 시점:
  세션이 끝나기 전, "HANDOFF.md 만들어줘"라고 요청
  → Claude가 현재 컨텍스트를 분석하고 인계 문서 작성

로드 시점 (두 가지 방법):
  방법 1: 수동 — "HANDOFF.md 읽고 이어서 해줘"
  방법 2: 자동 — CLAUDE.md에 다음을 추가:
          "HANDOFF.md가 있으면 세션 시작 시 자동으로 읽을 것"
          → Claude가 세션 시작 시 파일 존재 확인 후 자동 로드

작업 완료 후:
  HANDOFF.md를 삭제하거나, "완료" 표시로 갱신
```

**핵심**: /compact가 세션 내 기억 관리라면, HANDOFF.md는 **세션 간 기억 관리**다.
같은 원칙(핵심만 보존, 노이즈 버림)을 다른 시간 스케일에 적용한 것.

### 세션 이어가기: -c와 -r

HANDOFF.md를 쓰지 않아도 되는 가벼운 방법도 있다:

```bash
claude -c          # 마지막 세션을 이어서 시작
claude -r          # 최근 대화 목록에서 선택
```

`-c`는 마지막 대화의 컨텍스트를 그대로 불러온다.
`-r`은 과거 대화 목록을 보여주고 선택할 수 있게 한다.

```
가벼운 이어가기  → claude -c (컨텍스트 자동 복원)
의도적 인계     → HANDOFF.md (핵심만 선별 전달)
완전히 새 시작  → claude (빈 컨텍스트)
```

### 대화 포킹: 같은 출발점, 다른 방향

때때로 "방식 A와 방식 B 중 뭐가 나을까?"를 실험하고 싶다.
하지만 한 세션에서 A를 시도하면 컨텍스트가 A로 오염되어 B를 공정하게 시도할 수 없다.

```
대화 포킹 패턴:

세션 1: 구현 방식 A (Redux)
  → "인증 상태를 Redux로 관리해줘"
  → 결과 확인

세션 2: 구현 방식 B (Zustand)
  → claude -c  (세션 1의 컨텍스트 복원)
  → "아까 작업 취소하고, Zustand로 같은 것 구현해줘"
  → 결과 비교

또는 더 깨끗하게:
  HANDOFF.md에 "A와 B를 비교 중"이라고 적고
  두 개의 새 세션에서 각각 독립 실행
```

이것은 git branch의 정신과 같다: **같은 출발점에서 다른 경로를 탐색하고 비교한다.**

## Harness 관점: Context 레이어

Harness Engineering의 핵심 통찰:

```
Harness = Tools + Knowledge + Context + Permissions

Context 관리 = 에이전트에게 클린 메모리를 주는 것
```

컨텍스트가 꽉 차면 Claude의 **집중력이 흩어진다.**
관련 없는 오래된 내용들이 새로운 판단을 방해한다.

좋은 Harness 엔지니어는 컨텍스트를 능동적으로 관리한다:
- Micro compact: 조용히 정리
- Auto compact: 임계값에서 자동 정리
- Manual compact: 전략적으로 정리

**Claude가 일하기 좋은 작업 환경을 유지하는 것.**
이것이 컨텍스트 관리의 본질이다.
