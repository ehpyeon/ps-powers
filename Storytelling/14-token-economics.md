## "이 청구서가 맞아?"

다음 날 아침, API 사용량 대시보드를 열었다.

```
일일 비용 리포트:
  backend-agent  (Opus):  $47.20   (입력 280K + 출력 45K 토큰)
  frontend-agent (Opus):  $38.50   (입력 220K + 출력 40K 토큰)
  qa-agent       (Opus):  $29.80   (입력 170K + 출력 30K 토큰)
  ─────────────────────────────────
  합계:                   $115.50 / 하루
```

소연에게 이 비용을 보고할 수 있을까? 매일 $115?

지민은 자문했다. *"qa-agent한테 Opus가 정말 필요했을까?"*

## 모델 선택: 적재적소

Claude에는 세 가지 모델이 있다. 각각 능력과 비용이 다르다.

```
모델 비교 (대략적):

Opus 4.6  — 가장 똑똑함, 가장 비쌈
  강점: 복잡한 아키텍처 결정, 깊은 추론, 어려운 버그
  비용: $$$$
  속도: 느림

Sonnet 4.6 — 균형형
  강점: 일반 개발 작업, 코드 작성, 리팩토링
  비용: $$
  속도: 보통

Haiku 4.5  — 가장 빠름, 가장 저렴
  강점: 간단한 분석, 패턴 추출, 경량 작업
  비용: $
  속도: 빠름
```

### 모델 라우팅 전략

모든 작업에 Opus를 쓸 필요가 없다:

```
작업별 최적 모델:

Opus가 필요한 경우:
  → 복잡한 아키텍처 결정
  → 멀티파일 리팩토링 (파일 간 의존성 추론)
  → 미묘한 보안 취약점 분석
  → 새로운 시스템 설계

Sonnet으로 충분한 경우:
  → 일반적인 기능 구현
  → 코드 리뷰
  → 테스트 작성
  → 문서 생성

Haiku가 적합한 경우:
  → 로그 분석, 패턴 추출
  → 단순한 파일 변환
  → 백그라운드 모니터링
  → 코드 포맷팅 검사
```

## 에이전트 팀의 비용 최적화

지민은 팀 구성을 다시 설계했다:

```
최적화 전 (하루 $115.50):
  backend-agent  → Opus    $47.20
  frontend-agent → Opus    $38.50
  qa-agent       → Opus    $29.80

최적화 후 (하루 ~$35):
  backend-agent  → Sonnet  ~$15    (일반 구현은 Sonnet으로 충분)
  frontend-agent → Sonnet  ~$12    (UI 구현도 Sonnet)
  qa-agent       → Haiku   ~$5     (테스트 실행과 결과 분석)
  architect      → Opus    ~$3     (아키텍처 결정 시에만 호출)
  ─────────────────────────────────
  합계:                    ~$35 / 하루 (70% 절감)
```

**핵심: Opus는 "자문 역할"로만 쓴다.** 일상적인 구현은 Sonnet에게, 경량 작업은 Haiku에게.

### Agent 정의에서 모델 지정

```yaml
---
name: code-reviewer
description: 코드 리뷰 전문가
model: sonnet
tools: ["Read", "Grep", "Glob"]
---
```

```yaml
---
name: log-analyzer
description: 로그 분석 및 패턴 추출
model: haiku
tools: ["Read", "Grep", "Bash"]
---
```

`model` 필드 한 줄로 에이전트의 비용을 제어한다.

## 토큰 경제: 입력과 출력

비용을 이해하려면 토큰이 어디서 소비되는지 알아야 한다:

```
토큰 소비 구조:

입력 토큰 (Input):
  system prompt     ~500 토큰     (CLAUDE.md, Rules)
  messages[]        ~50,000 토큰   (대화 히스토리)
  tool_results      ~30,000 토큰   (파일 읽기, 명령 출력)
  ──────────────────────────────
  총 입력:          ~80,000 토큰

출력 토큰 (Output):
  Claude의 응답     ~1,000 토큰
  tool_use 호출     ~200 토큰
  ──────────────────────────────
  총 출력:          ~1,200 토큰
```

**입력이 출력보다 압도적으로 많다.** 비용 절감의 열쇠는 **입력 토큰을 줄이는 것**이다.

### 입력 토큰 절감 전략

```
전략 1: Micro Compact (Ch10)
  → 오래된 tool_result를 플레이스홀더로 교체
  → 절감: 턴당 ~3,000 토큰

전략 2: Skills On-demand (Ch04)
  → 필요할 때만 지식 로드, 시스템 프롬프트는 가볍게
  → 절감: 세션당 ~5,000 토큰

전략 3: Subagent 격리 (Ch06)
  → 탐색 노이즈를 부모 컨텍스트에서 차단
  → 절감: 작업당 ~20,000 토큰

전략 4: 전략적 /compact (Ch10)
  → 50회 툴 콜 후 수동 압축
  → 절감: ~50,000 토큰 (대화 초기화)

전략 5: 모델 라우팅
  → 저비용 모델로 경량 작업 처리
  → 절감: 작업당 60-80%
```

| 신호 | 원인 | 해결 |
|------|------|------|

## 노력 압축 테이블

비용만 보면 안 된다. **절약된 시간**도 봐야 한다.

| 작업 유형 | 사람 팀 | Claude Code | 압축률 |
|----------|---------|-------------|--------|
| 보일러플레이트 생성 | 2일 | 15분 | ~100x |
| 테스트 작성 | 1일 | 15분 | ~50x |
| 기능 구현 | 1주 | 30분 | ~30x |
| 버그 수정 + 회귀 테스트 | 4시간 | 15분 | ~20x |
| 아키텍처 설계 | 2일 | 4시간 | ~5x |

아키텍처 설계는 압축률이 낮다. **깊은 추론이 필요한 작업은 여전히 시간이 걸린다.** 하지만 보일러플레이트, 테스트, 반복 구현 — 이런 작업에서 Claude Code의 가치는 압도적이다.

$35/일로 시니어 엔지니어의 반복 작업을 대체한다면? ROI는 자명하다.

## 성찰: 비용은 제약이 아니라 설계 도구

소연에게 비용 리포트를 보여줬다.

```
최적화 전: $115/일 × 20일 = $2,300/월
최적화 후: $35/일 × 20일  = $700/월
```

소연: "이 정도면 시니어 개발자 월급의 10분의 1이야. 충분히 합리적이네."

비용을 의식하는 것은 **더 좋은 Harness를 만드는 것과 같다.** 모델 라우팅, 컨텍스트 압축, 지식 온디맨드 로딩 — 이 모든 기법이 비용 절감이면서 동시에 **품질 향상**이기도 하다.

컨텍스트에 노이즈가 적으면 Claude가 더 정확하게 판단한다. 적절한 모델을 쓰면 불필요한 대기 시간이 줄어든다.

**효율적인 Harness = 저렴한 Harness = 좋은 Harness.**
