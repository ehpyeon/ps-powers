# Skills — 반복을 패키징하다

> **이전 이야기에서**: 지민은 Hooks로 Claude의 행동을 제어하게 됐다.
> main에 직접 push하는 사고는 다시 일어나지 않았다. 파일 수정 후 lint도 자동으로 돌아갔다.
> 그런데 새로운 종류의 피로감이 쌓이고 있었다...

> Harness = Tools + **Knowledge** + Context + Permissions
>
> Skills는 **반복되는 Knowledge를 On-demand로 패키징**한다.
> *"Load knowledge when you need it, not upfront"*

---

## 하루에 10번 같은 말

지민은 하루에 커밋을 10번 했다. 매번 같은 지시를 타이핑했다.

```
"스테이지된 변경사항 보고, 컨벤셔널 커밋 형식으로 커밋해줘.
타입은 feat/fix/refactor/docs/test/chore 중 하나. 50자 이하. 한국어."
```

열 번. 매일. 아침에는 정확하게 지시했지만, 저녁에는 "그냥 커밋해줘"로 줄어들었다.
Claude의 커밋 품질이 아침과 저녁에 달랐다. **사람의 일관성에 한계**가 있었기 때문이다.

그리고 또 다른 문제가 있었다.

CLAUDE.md로 항상 필요한 Knowledge가 생겼다.
그런데 개발자들이 더 많은 것을 넣기 시작하면서 문제가 드러났다.

모든 Knowledge를 CLAUDE.md(시스템 프롬프트)에 넣으면:

```
시스템 프롬프트 = 프로젝트 정보 (500 토큰)
               + Git 워크플로우 (1000 토큰)
               + 코드 리뷰 체크리스트 (2000 토큰)
               + 테스트 가이드라인 (1500 토큰)
               + 배포 절차 (2000 토큰)
               + 보안 가이드 (1800 토큰)
               + ...
               = 총 15,000 토큰
```

**문제 1: 컨텍스트 낭비**
`/commit`을 실행할 때 배포 절차 2000 토큰은 필요 없다.
`/deploy`를 실행할 때 코드 리뷰 체크리스트 2000 토큰은 필요 없다.
모든 지식을 항상 들고 다니는 건 낭비다.

**문제 2: 반복 지시의 비효율**
매번 같은 지시를 내리는 것과는 다른 문제:
"이 워크플로우를 실행할 때는 이 지식이 필요하다"는 것을 어떻게 자동화할까?

---

## 해결책: Two-Layer Knowledge Loading

핵심 패턴:

```
Layer 1 (항상 활성화 — 시스템 프롬프트):
┌─────────────────────────────────────┐
│ 사용 가능한 스킬:                    │
│   - git: Git 워크플로우 헬퍼         │  ~100 토큰/스킬
│   - code-review: 코드 리뷰 체크리스트│
│   - deploy: 배포 절차               │
└─────────────────────────────────────┘
             ↓ Claude가 load_skill("git") 요청
Layer 2 (필요할 때만 — tool_result):
┌─────────────────────────────────────┐
│ <skill name="git">                  │
│   ## Git 워크플로우                  │  ~2000 토큰
│   1. 브랜치 전략: ...               │
│   2. 커밋 형식: ...                 │
│ </skill>                            │
└─────────────────────────────────────┘
```

**Layer 1**: 어떤 Knowledge가 있는지만 알려준다 (이름 + 한 줄 설명) — 저렴
**Layer 2**: 실제로 필요할 때 전체 내용을 로드한다 — 필요할 때만

10개 스킬 × 100토큰 = 1,000 토큰 (레이어 1 항상)
vs
사용된 스킬 × 2,000 토큰 = 평균 2,000-4,000 토큰 (레이어 2 필요 시)

총 소비 토큰이 크게 줄었다.

## Claude Code에서 Skills = 슬래시 커맨드

Two-Layer 패턴이 Claude Code에서는 **슬래시 커맨드**로 구현됐다.

`.claude/skills/commit/SKILL.md` 파일 하나를 만들면 `/commit` 커맨드가 생긴다.

```
두 가지 Knowledge 로딩 방식:
1. CLAUDE.md/Rules = 항상 활성화된 배경 지식
2. Skills = 명시적으로 요청될 때 로드되는 전문 지식
```

차이점:
- CLAUDE.md: "이 프로젝트는 TypeScript야" → 항상 필요
- Skill: "컨벤셔널 커밋 형식 전체 규칙" → `/commit`할 때만 필요

---

## SKILL.md의 구조

실제 프로젝트의 `/commit` 스킬 전체를 살펴보자:

```markdown
---
name: commit
description: 스테이지된 변경사항을 분석하고 컨벤셔널 커밋 메시지를 생성하여 커밋한다.
---

스테이지된 변경사항을 분석하고 컨벤셔널 커밋을 생성한다.

## 실행 단계

1. `git status`와 `git diff --staged`로 스테이지된 변경사항 확인
2. 변경사항이 없으면 사용자에게 알리고 종료
3. 변경사항의 목적과 범위를 파악하여 커밋 메시지 초안 작성
4. 아래 형식으로 커밋 메시지 생성
5. `git commit -m "..."` 실행

## 커밋 메시지 형식

<type>: <설명>

<선택적 본문 — 왜 이 변경을 했는지>

**타입 선택 기준:**
- `feat` — 새로운 기능
- `fix` — 버그 수정
- `refactor` — 기능 변경 없는 코드 개선
- `docs` — 문서만 변경
- `test` — 테스트 추가/수정
- `chore` — 빌드 설정, 의존성 등 기타

## 규칙
- 설명은 한국어로, 50자 이하
- 본문은 "what"이 아닌 "why"
- `--no-verify` 절대 사용 금지

## $ARGUMENTS 처리
인자가 있으면 추가 컨텍스트로 활용한다.
예: `/commit 로그인 버그 관련` → 해당 맥락을 반영한 메시지 생성
```

이것이 Layer 2 전체다. `/commit`을 호출할 때만 이 ~45줄이 Claude의 컨텍스트에 들어온다.

> **💡 Tip:** Skill은 파일이 아니라 **폴더**다. `.claude/skills/commit/` 폴더 안에 SKILL.md 외에 스크립트, 데이터, 예시 파일을 함께 넣을 수 있다.

> **❌ Anti-pattern:** 당연한 것을 Skills에 쓰지 마라. Claude가 이미 알고 있는 것(언어 문법, 프레임워크 기본)을 반복하면 컨텍스트만 낭비한다. **비기본 동작, 팀의 규칙, 자주 틀리는 부분**에 집중하라.

> **💬** "가장 가치 높은 콘텐츠는 Gotchas 섹션이다. 처음에는 비어 있어도 좋다. 시간이 지나면서 실패 포인트를 계속 추가하라. 이것이 가장 높은 신호 대 잡음 비율을 가진다." — Thariq, Anthropic Engineer

## 9가지 스킬 카테고리

"어떤 스킬을 만들어야 할까?" 지민은 `/commit`을 만들었지만, 다른 건 뭘 만들어야 할지 몰랐다.

Anthropic 팀의 경험에 따르면, 모든 스킬은 9가지 카테고리 중 하나에 속한다:

| 카테고리 | 설명 | 예시 |
|---------|------|------|
| **Library & API Reference** | 자주 쓰는 라이브러리의 비표준 사용법 | Stripe API 가이드, ORM 규칙 |
| **Product Verification** | 제품 기능을 자동으로 검증 | 회원가입 플로우 테스트, 결제 E2E |
| **Data Fetching & Analysis** | 데이터 조회와 분석 자동화 | DB 쿼리, 대시보드 생성 |
| **Business Process** | 반복적인 비즈니스 워크플로우 | 스탠드업 포스트, 주간 리포트 |
| **Code Scaffolding** | 프로젝트 규칙에 맞는 보일러플레이트 | 컴포넌트 템플릿, API 라우트 생성 |
| **Code Quality & Review** | 팀 규칙 기반 코드 검증 | 코드 리뷰, 스타일 검사 |
| **CI/CD & Deployment** | 빌드, 테스트, 배포 자동화 | PR 관리, 배포 파이프라인 |
| **Runbooks** | 장애 대응, 디버깅 절차 | 서비스 디버깅, 온콜 대응 |
| **Infrastructure Ops** | 인프라 유지보수 자동화 | 의존성 업데이트, 비용 조사 |

처음에는 1-2개 카테고리만 시작하고, 프로젝트가 성장하면서 확장하면 된다.

## $ARGUMENTS: 동적 Knowledge

Skills는 정적인 지식만 담는 게 아니다.
`$ARGUMENTS`로 동적으로 만들 수 있다.

```
/commit 로그인 버그 수정
         ↑
         $ARGUMENTS = "로그인 버그 수정"
```

`/compact` 스킬은 `$ARGUMENTS`로 세 가지 모드를 분기한다:

```markdown
## $ARGUMENTS 확인:
- 인자 없음 → 상태 분석 후 권장 여부 알려줌
- "force" 또는 "지금" → 즉시 /compact 실행
- "check" → 현재 상태만 보고
```

같은 스킬 하나로 `/compact`, `/compact force`, `/compact check`가 모두 다르게 동작한다.

## CLAUDE.md vs Rules vs Skills — 무엇을 어디에?

```
이 Knowledge를 어디에 넣을까?

항상 필요한 정보 (프로젝트 개요, 기술 스택):
  → CLAUDE.md

항상 필요한 규칙 (코딩 스타일, 보안 정책):
  → .claude/rules/*.md

명시적으로 호출될 때만 필요한 것 (/commit, /plan):
  → .claude/skills/*/SKILL.md

판단 기준:
  "지금 이 세션에서 필요한가?" → CLAUDE.md / Rules
  "특정 워크플로우를 실행할 때만 필요한가?" → Skills
```

## 반복 제거: 같은 말을 계속 타이핑하는 문제

Skills가 해결하는 표면적 문제:
매번 같은 지시를 타이핑하는 낭비.

```
# 스킬 없이
나: 스테이지된 변경사항 보고, 컨벤셔널 커밋 형식으로 커밋해줘.
    타입은 feat/fix/refactor/docs/test/chore 중 하나. 50자 이하. 한국어.
    (하루에 10번)

# 스킬 있음
나: /commit
    (하루에 10번)
```

그러나 더 깊은 가치는 **일관성**이다.
사람은 피곤하면 지시를 대충 내린다.
Skill은 항상 같은 품질의 Knowledge를 주입한다.

## Skills가 성숙하면서 생긴 문제

Skills로 워크플로우가 패키징됐다.
그런데 워크플로우가 복잡해질수록 다른 문제가 드러났다.

`/plan`을 실행하면 Claude가 계획을 짠다.
그런데 계획을 짜면서 Claude의 메인 컨텍스트에 노이즈가 쌓였다.

```
메인 대화 컨텍스트:
- 이전 3시간 동안의 구현 내용
- 잡담
- 계획 수립 과정의 파일 읽기 30개
→ 이 모든 노이즈를 들고 코드를 작성?
```

*"계획 수립은 깨끗한 컨텍스트에서 하고 싶다.
코드 리뷰는 코드만 보는 전문가에게 맡기고 싶다."*

이것이 Harness의 **Context Isolation 레이어**에 대한 수요였다.

*"단계가 많은 작업에서 에이전트가 길을 잃는다. Skills로 워크플로우를 패키징했지만, 10단계짜리 구현은 어떻게 관리하나?"*

→ **[Ch05. Task System — 어디까지 했어?](./08-todo-task-system.md)**
